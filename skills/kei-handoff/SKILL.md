# Kei Handoff

現在のタスクで確定した引継ぎ情報を、自然な日本語で読める自己完結した実装指示へ変換する。

## 責務

- リポジトリを調査しない。
- 提供された事実、決定、制約を変更、補完、再判断しない。
- 実装、ファイル編集、commit、タスクやworkspaceの作成、Superset起動を行わない。
- リポジトリごとに実装指示を1つ作る。
- 起動方式、推論レベル、workspace ID、terminal commandは実装指示に入れない。実行skillが管理する。
- 引継ぎ情報が不足している場合は、実装判断を発明せず準備未完了と返す。

## 引継ぎ準備

実装指示を作る前に、現在のタスクから次を確認する。

- 対象リポジトリ
- 問題または期待する結果
- 成功条件
- 判断の正本、または明示的なユーザー判断（挙動が曖昧な場合）
- 制約と対象外。特にない場合はその事実
- 実装を止める未確定事項が0件
- 受け入れるリスクと、実装担当に調査を任せる未確定事項（存在する場合）

次は引継ぎ準備の必須条件にしない。実装担当エージェントがリポジトリを調査して決める。

- 根本原因
- 実装方式
- 主な対象ファイル
- 詳細な実装手順
- 実行するテストコマンド

不明点の答えによってユーザーに見える挙動、APIやDBの契約、対象範囲、成功条件が変わる場合は、実装を止める未確定事項として扱う。これが残る場合は実装指示を作らず、次の形式で確認を1つだけ返す。

```markdown
ハンドオフの準備が完了していません。

| # | 不足している判断 | 実装を止める理由 | 確認したいこと |
|---:|---|---|---|
| 1 | <未確定事項> | <影響> | <質問を1つ> |
```

## 変換ルール

- 引継ぎ情報の問題、期待結果、判断、制約、事実を維持する。
- 同じ内容を複数の節へ重複させない。
- 実装指示の見出し、説明、指示、報告項目など、自然言語部分はすべて分かりやすい日本語で書く。
- 提供された自然言語が日本語以外の場合は、意味や判断を変えず日本語へ訳す。
- identifier、path、command、error string、model slugなど、正確さが必要な技術リテラルは原文を保持する。
- 専門用語が必要な場合は、普通の日本語を先に書き、必要な時だけ技術用語を添える。
- local pathや`cd`は、提供された情報または実行基盤が必要とする場合だけ含める。
- 実装用の指示と読み取り専用の指示で、実行ルールを分ける。
- ユーザーの明示条件を以下の既定ルールより優先する。

## 実装担当エージェントへの指示

引継ぎ情報を一度だけ記載し、その後に共通の実行ルールを一度だけ付ける。

```text
あなたは実装担当エージェントです。別のハンドオフ用プロンプトは作らず、このリポジトリで今すぐ実装してください。

実装の前提情報:
<現在のタスクで確定した情報を、意味や判断を変えず日本語で記載する>

実行ルール:
- 実装の前提情報をこのタスクの判断の正本として扱い、問題、期待結果、成功条件、制約、対象外を守る。
- 編集前にリポジトリを調査し、現在の動作、根本原因、関連する処理経路、既存の実装パターン、必要な検証を確認する。
- 作業開始時にブランチ、HEAD、`git status --short`を記録する。
- 対象タスクに必要な、最小で一貫した変更を行う。追加ファイルの変更が必要になった場合は最小限に留め、対象が広がったファイルと理由を報告する。
- 明示的に保護されたファイル、ユーザー所有の変更、作業開始前からある未commit変更を変更、削除、commitしない。
- 実装の前提情報またはユーザーが明示的に許可していないpush、PR作成、deploy、本番変更、外部サービスへの書き込みを行わない。
- 現在のコード、テスト、ドキュメントと実装の前提情報が衝突し、どれを正とするか決められない場合は停止して証拠を報告する。実装の前提情報から古いと判断できる期待値は、最小修正して続行する。
- 作業停止条件は、明示制約との衝突、機密情報、未許可の外部変更、破壊的操作、資格情報不足、検証不能、または現在のコードと実装の前提情報だけでは決められない製品判断に限る。

検証と完了処理:
- 成功条件を証明するために必要な範囲の検証を特定して実行する。
- 実装と検証の後、この実装差分に対して`$codex-review`を最終レビューとして使う。
- 対応が必要な指摘が今回の実装範囲に直接含まれる場合は、最小修正し、指摘がなくなるまで必要な検証とレビューを繰り返す。
- 実装の前提情報またはユーザーがcommit禁止を明示しない限り、検証とレビュー後に、このタスクで意図した変更だけを含むローカルcommitを作る。
- 開始時に未commit変更がなければ、終了時にも残さない。開始時に未commit変更があった場合はその状態を維持し、このタスク由来の未commit差分を残さない。

完了報告:
- ブランチ、基準commit、変更したファイル
- 当初の対象外から追加で変更したファイルと理由
- commit SHAとメッセージ、またはcommitを省略した理由
- 検証結果
- 最終レビュー結果
- 最終的な`git status --short`と開始時の状態との差
- 実装担当に調査を任せた未確定事項の解決内容、受け入れたリスク、作業を止めた項目
- 最終報告の末尾に`実装対象:`を置き、実装の前提情報にある問題または期待結果を再掲する。

停止条件:
成功条件を満たし、必要な範囲の検証、最終レビュー、必要なローカルcommit、作業ツリー確認を完了したら停止する。作業停止条件がある場合は変更範囲を広げず、証拠と必要な判断を報告して停止する。
```

## 読み取り専用エージェントへの指示

調査、診断、レビュー用の指示には実装用の実行ルールを付けない。

```text
あなたは読み取り専用の調査担当エージェントです。次の依頼をこのタスクに依存せず実行してください。

対象リポジトリ:
<owner/repo>

依頼内容:
<提供された情報を、意味を変えず日本語で記載する>

制約:
- ファイル編集、commit、push、PRレビュー投稿、外部サービスへの書き込みを行わない。
- 確認済みの事実、証拠からの推論、仮定、未確認事項を混ぜない。
- 対象差分または対象状態を明記し、観測できる証拠を優先する。

報告内容:
- 結論
- 根拠
- 不確実な点または作業停止条件
- 次に必要な最小対応
```

## 出力

対象が1リポジトリなら実装指示を1つだけ返す。複数リポジトリならリポジトリごとに分ける。

実装指示以外の説明は、どの引継ぎ情報を何の指示へ変換したかを示す自然な日本語1文だけにする。
